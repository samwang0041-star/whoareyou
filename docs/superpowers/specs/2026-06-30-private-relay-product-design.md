## Problem Statement

Build a new product branch on top of the existing UNKNOWN/OpenClaw infrastructure.

The product is a one-to-one hidden identity relay disguised as an AI entry point:

1. A opens the site and gets an entry QR.
2. A scans and binds the entry.
3. The site gives A a private share link.
4. A sends that link to B.
5. B opens the link, gets their own entry QR, scans, and binds.
6. Once both sides are bound, messages sent through the WeChat/OpenClaw AI entry are forwarded between A and B.
7. Either side can send `/断开`.
8. After disconnect, the relationship is gone. They cannot talk again through that link. To reconnect, someone must reopen the site, scan again, share again, and establish a new link.

This is not random matching. It is not a community. It is not anonymous social networking. It is a small, deliberate relay toy: an AI-looking entrance where the other side is actually a person, with identity hidden by the server.

## What Makes This Cool

The product has one strong trick: both people think they are entering an AI-like channel, but the real magic is that the channel becomes a private human-to-human wire.

The charm is in how little it does:

- No profile.
- No friend relationship.
- No searchable history.
- No matching pool.
- No permanent contact.
- One command: `/断开`.

It should feel like someone found a strange bug in an AI entrance and chose not to fix it immediately.

## Constraints

- Use the existing Next.js, Prisma, PostgreSQL, OpenClaw QR, OpenClaw getupdates, and sendmessage foundation.
- Do not store readable chat history.
- Do not add nicknames, avatars, phone numbers, or identity fields.
- Accept message loss. Relay messages do not need durable retry.
- Server may keep routing state, encrypted provider delivery refs, operational counters, and cleanup metadata.
- The server should not use the existing random matching pool.
- The server should not create one-hour reminders, cooldowns, reports, or rematch behavior for this branch.
- The user-facing command surface is only `/断开`.
- A share link is single-use for B and expires.
- A QR is short-lived and refreshable.
- A closed or expired relationship cannot be resumed.

## Premises

1. The product value is the private relay mechanism, not discovery. A chooses who gets the link.
2. Because A already shares the link to B, abuse controls can be much smaller than the random-matching product.
3. Losing an occasional message is acceptable if it lets the server avoid becoming a chat database.
4. The existing OpenClaw credential and provider-ref infrastructure is valuable and should be reused.
5. The existing random matching state machine is intentionally the wrong abstraction for this product.

## Approaches Considered

### Approach A: Reuse Current User/Connection Flow

Summary: Keep the current `User`, `Connection`, `MessageOutbox`, and inbound handler, but create a connection from an invite instead of random matching.

Effort: S
Risk: Med

Pros:

- Fastest path to a demo.
- Reuses tested outbox, provider ref, admin, and fake callback flow.
- Minimal schema work.

Cons:

- Pulls in one-hour timers, cooldown, report, pair-block, waiting, and matching language unless carefully disabled.
- Relay messages still pass through `MessageOutbox`, which conflicts with the stated "no local DB message forwarding" principle.
- Product behavior becomes harder to reason about because old commands still exist.

Reuses:

- `src/adapters/openclaw-*`
- `src/domain/identity.ts`
- `src/workers/outbox.ts`
- `Connection`

### Approach B: Separate Relay Domain, Reuse Provider Infrastructure

Summary: Add relay-specific invite/session tables and a relay inbound handler. Reuse OpenClaw QR creation, QR status, encrypted provider refs, getupdates polling, and sendmessage runtime. For user relay messages, send directly best-effort instead of storing them in `MessageOutbox`.

Effort: M
Risk: Low/Med

Pros:

- Keeps the product small and legible.
- Preserves existing provider hardening without inheriting random matching behavior.
- Supports "no readable chat logs" and "message loss acceptable" cleanly.
- Easy to test with fake provider by asserting no relay message body is persisted.

Cons:

- Requires new schema and a mode switch in the inbound worker.
- Direct send means provider outages can drop messages.
- Admin metrics need new relay-specific counters instead of relying on current connection dashboards.

Reuses:

- `OpenClawBotSession`
- `UserProviderRef` or a relay-specific encrypted provider ref model
- `src/adapters/openclaw-weixin-entry.ts`
- `src/adapters/openclaw-weixin-runtime.ts`
- `src/web/rate-limit.ts`
- `src/workers/openclaw-updates.ts` polling loop

### Approach C: Fully Ephemeral In-Memory Relay

Summary: Keep all invite and routing state in memory. Database stores nothing except optional aggregate counters.

Effort: S
Risk: High

Pros:

- Closest literal interpretation of "server only forwards."
- No cleanup burden for relationship state.
- Very little data remains after process exit.

Cons:

- Breaks on deploys, restarts, multiple processes, and worker/web separation.
- OpenClaw polling and sending already need durable provider session state.
- A share link can become invalid for reasons users cannot understand.

Reuses:

- Web QR UI only.

## Recommended Approach

Choose Approach B: Separate Relay Domain, Reuse Provider Infrastructure.

It is the smallest architecture that respects the product's taste. The server keeps only what is needed to route the current relationship and operate OpenClaw. It does not become a chat database, and it does not drag the random matching product into a direct-share product.

## Product Flow

### Web Flow

`GET /`

- Shows a minimal entry.
- Primary action: create a private relay.
- Copy should explain function, not lore:
  - "生成一个入口，把它交给一个人。"
  - "对方接入后，你们可以在微信 AI 入口里说话。"
  - "发 `/断开`，关系立即消失。"

`POST /api/relay/invites`

- Creates a relay invite.
- Issues A's QR session.
- Returns:
  - `inviteId`
  - `aQr.imageSrc`
  - `aQr.statusUrl`
  - `expiresAt`

`GET /api/relay/invites/:inviteId/status`

- Polls whether A has scanned.
- After A is bound, returns the B share link.

`GET /r/:shareSlug`

- B opens A's private share link.
- If valid and unused, creates or returns B's QR session.
- If expired, consumed, or closed, renders a dead-link state.

`GET /api/relay/invites/:inviteId/b/status`

- Polls whether B has scanned.
- Once B is bound, returns `connected`.

### WeChat/OpenClaw Flow

Inbound message handling is mode-specific:

- `PRODUCT_MODE=unknown`: current random matching behavior.
- `PRODUCT_MODE=private_relay`: new relay behavior.

For `private_relay`:

1. Parse inbound provider user and text.
2. Resolve the provider user to an active relay participant.
3. If text is `/断开`, close the relay and best-effort notify the other side.
4. If both sides are connected, directly call OpenClaw sendmessage to the peer.
5. If the peer is missing, expired, or closed, reply with a short no-active-relay message.
6. Do not insert user message text into `MessageOutbox`, `Echo`, or any transcript table.

## Relay State Model

### RelayInvite

Fields:

- `id`
- `shareSlug`
- `state`: `created | a_qr_issued | a_bound | b_qr_issued | connected | closed | expired`
- `aBotSessionId`
- `bBotSessionId`
- `connectionId` or `relayConnectionId`
- `expiresAt`
- `createdAt`
- `updatedAt`
- `connectedAt`
- `closedAt`
- `closeReason`: `disconnected | expired | provider_expired`

Indexes:

- `shareSlug` unique
- `state, updatedAt`
- `expiresAt`

### RelayParticipant

Fields:

- `id`
- `inviteId`
- `role`: `a | b`
- `provider`: `openclaw-weixin`
- `providerUserHash`
- `providerUserIdCiphertext`
- `latestContextTokenCiphertext`
- `botSessionId`
- `joinedAt`
- `lastInboundAt`
- `deletedAt`

Constraints:

- unique `inviteId, role`
- unique active `provider, providerUserHash` if one provider identity cannot join multiple active relays at the same time

Privacy rule:

- On close, delete or tombstone participant delivery refs after peer notification attempt.
- Keep only non-identifying aggregate counters for ops.

### RelayConnection

Fields:

- `id`
- `inviteId`
- `state`: `active | closing | closed`
- `startedAt`
- `closedAt`
- `closeReason`

No message table is attached to `RelayConnection`.

## Direct Relay Semantics

The relay path is intentionally best-effort:

1. Inbound arrives from A.
2. Server resolves B's current encrypted provider ref.
3. Server calls OpenClaw sendmessage to B immediately.
4. If send succeeds, no message body remains.
5. If send fails, record an operational error without raw message text.
6. Do not retry user text unless a later explicit product decision changes this.

Allowed persistence:

- message length
- provider status category
- error fingerprint
- participant role
- invite/connection ids
- timestamps

Forbidden persistence:

- raw message body
- normalized message body
- excerpts
- provider raw user id in logs
- B share link in logs

## Command Surface

Only one user-facing command is part of this branch:

```text
/断开
```

Behavior:

- Works from either side.
- Immediately closes the relay.
- Best-effort sends a short close message to the peer.
- Invalidates the share link and both participant routes.
- Deletes or clears active provider delivery refs after notification.

All other text is treated as relay content if connected.

## Copy Direction

The page should not over-explain the philosophy. The product is stronger when it feels functional and slightly strange.

Suggested minimal copy:

```text
UNKNOWN RELAY

生成一个入口，把它交给一个人。
对方接入后，你们会在同一个微信 AI 入口里相遇。

发 /断开，关系立即消失。
```

QR modal states:

- A QR waiting: "先用微信接入这个入口。"
- A bound: "入口已经认得你。把这个链接交给一个人。"
- B waiting: "对方接入后，关系才会建立。"
- Connected: "已经接通。回到微信说话。"
- Expired: "这张入口已经失效。重新生成。"

WeChat replies:

- A bound: "我在。把页面上的链接交给那个人。"
- B connected: "接通了。你们现在可以说话。发 `/断开`，关系会消失。"
- Peer disconnected: "对方断开了。这段关系已经消失。"
- No active relay: "这里没有正在连接的人。重新打开网站，生成新的入口。"

## Admin/Monitoring

For this branch, admin should focus on operational health, not social moderation:

- invites created per hour
- A bound count
- B bound count
- active relay count
- closed relay count
- expired invite count
- relay send success/failure count
- provider send latency p50/p95
- OpenClaw QR creation failures
- OpenClaw status polling failures
- getupdates failures
- cleanup deletes
- database row counts for relay tables

Admin must not expose:

- raw provider ids
- share slugs
- raw message text
- encrypted credential blobs

## Security And Abuse

- Rate limit invite creation by IP.
- Rate limit QR status polling by IP.
- Share slug must be unguessable, at least 128 bits of entropy.
- B link is single-use and expires.
- A cannot create unlimited active invites from one browser/IP in a short window.
- A provider user cannot hold unlimited active relay roles.
- Clamp inbound text length before sending.
- Do not reflect inbound text into web HTML.
- Do not include share slugs or provider refs in logs.
- Cleanup expired invites and cleared refs on a short cadence.
- Production must keep existing `PROVIDER_MODE=openclaw` and fake-provider protections.

## Success Criteria

- A can create a relay invite and scan A QR.
- After A scan, the web page shows a share link for B.
- B can open the share link and scan B QR.
- After B scan, both sides receive connected guidance in WeChat.
- A normal message from A is sent to B.
- A normal message from B is sent to A.
- Sending `/断开` from either side closes the relay.
- After close, further messages are not forwarded.
- The same share link cannot create a second relationship.
- Raw chat message text is not persisted in PostgreSQL after relay send.
- Relay messages do not create `MessageOutbox` rows.
- Existing UNKNOWN random matching flow remains untouched on `main`.

## Implementation Plan

1. Add `PRODUCT_MODE=unknown | private_relay` config with production validation.
2. Add relay schema tables and migration.
3. Add relay domain service:
   - create invite
   - attach A
   - issue B share link
   - attach B
   - close relay
   - resolve peer route
4. Add relay web APIs and pages:
   - `POST /api/relay/invites`
   - `GET /api/relay/invites/:id/status`
   - `GET /r/:shareSlug`
5. Add `handleRelayInbound(event)` and switch `openclaw-updates` by `PRODUCT_MODE`.
6. Add direct OpenClaw send helper that sends to a `RelayParticipant` without creating outbox rows.
7. Add cleanup for expired relay invites and cleared refs.
8. Add admin metrics for relay mode.
9. Add unit tests for `/断开` parsing and relay state transitions.
10. Add integration tests for fake provider A/B scan, relay, disconnect, expiry, and no message persistence.
11. Add Playwright flow for the web invite/share-link path.

## Distribution Plan

This is a web service branch. It can use the existing deployment model:

- Next.js web process
- OpenClaw updates worker
- scheduled cleanup worker
- PostgreSQL migrations

The existing outbox worker is not required for user relay messages in `private_relay` mode. If system notifications still use outbox, keep it explicit and ensure user text never enters it.

## Open Questions

- Product name for this branch: keep `UNKNOWN`, use `UNKNOWN RELAY`, or give it a separate name.
- Whether provider identity may participate in multiple active relays at once. Default recommendation: no.
- Whether to notify A inside WeChat when A has scanned but has not shared the link yet. Default recommendation: one short message only.
- Whether to deploy this at a separate domain/subpath or switch the main domain temporarily. Default recommendation: separate branch and separate deployment target until tested.

## What I Noticed

- The strongest product constraint is "pure forwarding, identity hidden, relationship disappears." That is more important than adding social controls.
- The product gets weaker when it explains too much. The interaction should carry the idea.
- The current codebase already solved the hard provider/security parts. The new work is mostly subtractive: avoid random matching, avoid timelines, avoid chat storage.
